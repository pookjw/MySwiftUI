# 1. 조사 주제

이번 자료찾기에서는 ARM 바이너리 분석, 디스어셈블, 역공학, 자동 리프팅, 역난독화 같은 주제와 관련된 한국어 논문들을 찾아보았다.  
처음에는 비슷한 말이 많아서 조금 헷갈렸는데, 찾아보니까 결국은 기계어 또는 바이너리 코드를 사람이 이해할 수 있게 바꾸거나 분석하는 연구들이라고 볼 수 있었다.

# 2. 논문 조사

## 1) 인공신경망 기계번역 및 어셈블리 언어 모델 기반 자동 바이너리 리프팅

서지사항: 또올가, 윤요섭, 정민찬, 임강빈, 「인공신경망 기계번역 및 어셈블리 언어 모델 기반 자동 바이너리 리프팅」, 정보보호학회논문지, 2025, pp.241-251.

요약: 이 논문은 어셈블리 코드를 LLVM IR 같은 중간 표현으로 자동 변환하는 방법을 다룬 논문이다. 그냥 규칙 몇 개로 바꾸는 방식이 아니라, 신경망 기계번역 방식이랑 어셈블리 언어 모델을 같이 사용한 점이 특징이라고 볼 수 있었다. 기존 방식보다 결과가 더 좋아졌다고 설명하고 있다.

느낀 점: 내가 보기에는 이 논문이 제일 최신 느낌이었고, 사람이 하나하나 해석하는 것 말고 자동으로 올리는 방향을 보여주는 것 같았다. 다만 이런 방법도 완전히 정확하다고 보기에는 아직 어려울 수도 있겠다고 생각했다.

인용 예시: 또올가 외(2025, pp.241-251)는 기존의 휴리스틱 기반 바이너리 리프팅 방식의 한계를 보완하기 위해 신경망 기반 자동 리프팅 방법을 제안하였다.

출처: <https://www.kci.go.kr/kciportal/ci/sereArticleSearch/ciSereArtiView.kci?sereArticleSearchBean.artiId=ART003196116>

## 2) ARM 바이너리 코드에 대한 데이터흐름분석 방법

서지사항: 엄기진, 조은선, 백준영, 「ARM 바이너리 코드에 대한 데이터흐름분석 방법」, 정보과학회논문지: 소프트웨어 및 응용, 2014, pp.387-395.

요약: 이 논문은 ARM 바이너리 코드에 대해서 디스어셈블을 하고, 그 다음 데이터흐름분석까지 연결해서 보는 방법을 설명한다. 단순히 명령어만 보는 것이 아니라 값이 어디서 오고 어디로 가는지를 같이 보는 게 중요하다는 내용이 핵심처럼 보였다. IDA Pro 기반으로 도구를 만들었다는 점도 눈에 들어왔다.

느낀 점: 그냥 한 줄 한 줄 번역하는 것보다, 레지스터 값이 어떻게 움직이는지 따라가는 게 더 중요하다는 걸 알 수 있었다. ARM 쪽 논문이라서 이번 조사 주제하고도 잘 맞는다고 느꼈다.

인용 예시: 엄기진 외(2014, pp.387-395)는 ARM 바이너리 분석에서 디스어셈블과 데이터흐름분석을 함께 수행하는 접근이 필요하다고 보았다.

출처: <https://www.kci.go.kr/kciportal/ci/sereArticleSearch/ciSereArtiView.kci?sereArticleSearchBean.artiId=ART001875535>

## 3) 어셈블리어 코드 기반의 메모리 오류 가능성 검출

서지사항: 김현수, 김병만, 배현섭, 정인상, 「어셈블리어 코드 기반의 메모리 오류 가능성 검출」, 정보처리학회논문지D, 2011, pp.35-44.

요약: 이 논문은 실행 파일을 역어셈블해서 나온 어셈블리 코드를 분석하고, 그걸 바탕으로 메모리 오류 가능성을 찾는 방법을 제안한다. 특히 null pointer access error 같은 문제를 찾는 데 초점을 맞추고 있었다. 명령어 전이도를 만든다는 부분이 좀 어려웠지만, 결국 프로그램 흐름을 보고 위험한 부분을 찾는 내용이라고 이해했다.

느낀 점: 역공학이 단지 코드를 읽는 것만이 아니라, 실제 오류를 찾는 데도 연결될 수 있다는 점이 인상적이었다. 분석 결과가 실용적인 쪽으로 이어질 수 있다는 생각이 들었다.

인용 예시: 김현수 외(2011, pp.35-44)는 역어셈블된 어셈블리 코드를 분석하여 메모리 사용 오류 가능성을 검출하는 방법을 제시하였다.

출처: <https://www.kci.go.kr/kciportal/ci/sereArticleSearch/ciSereArtiView.kci?sereArticleSearchBean.artiId=ART001532139>

## 4) 단편화된 실행파일을 위한 데이터 구조 역공학 기법

서지사항: 이종협, 「단편화된 실행파일을 위한 데이터 구조 역공학 기법」, 정보보호학회논문지, 2012, pp.615-619.

요약: 이 논문은 단편화된 실행파일에서는 기존 데이터 분석 방식이 잘 안 맞는 문제를 이야기하고, 그걸 개선하기 위한 데이터 구조 역공학 방법을 제시한다. 프로그램을 이해할 때 제어 흐름도 중요하지만 데이터 구조를 같이 봐야 한다는 점을 강조하고 있었다.

느낀 점: 처음에는 데이터 구조 역공학이라는 말이 조금 낯설었는데, 생각해 보니까 결국 변수나 구조체 비슷한 걸 복원하는 문제라고 볼 수 있었다. 프로그램 이해를 더 깊게 하려면 이런 부분도 꼭 필요하겠다고 느꼈다.

인용 예시: 이종협(2012, pp.615-619)은 소프트웨어를 이해하기 위해서는 제어 구조뿐 아니라 데이터 구조에 대한 역공학도 중요하다고 설명하였다.

출처: <https://www.kci.go.kr/kciportal/ci/sereArticleSearch/ciSereArtiView.kci?sereArticleSearchBean.artiId=ART001678581>

## 5) 트램폴린 코드 기반의 난독화 기법을 위한 역난독화 시스템

서지사항: 김민호, 이정현, 조해현, 「트램폴린 코드 기반의 난독화 기법을 위한 역난독화 시스템」, 정보보호학회논문지, 2023, pp.1043-1053.

요약: 이 논문은 난독화된 코드를 다시 분석하기 쉽게 풀어주는 역난독화 시스템에 대한 논문이다. 특히 트램폴린 코드 기반 난독화를 대상으로 하고 있어서 제어 흐름이 복잡하게 꼬이는 문제를 다룬다. 분석자가 원래 코드 흐름을 더 잘 보게 만드는 것이 목적이라고 이해했다.

느낀 점: 난독화는 그냥 코드를 보기 어렵게 만드는 기술 정도로만 알고 있었는데, 실제 분석에서는 이런 걸 다시 푸는 과정도 중요하다는 걸 알게 되었다. 역공학이 생각보다 방어 기술과도 많이 연결된다고 느꼈다.

인용 예시: 김민호 외(2023, pp.1043-1053)는 난독화된 제어 흐름을 복원하기 위한 역난독화 시스템의 필요성을 제시하였다.

출처: <https://www.kci.go.kr/kciportal/landing/article.kci?arti_id=ART003025963>

## 6) 프로토콜 역공학 연구 동향 및 최적의 악성코드 통신 메시지 분석환경 구축 연구

서지사항: 신강식, 정동재, 최민지, 조호묵, 「프로토콜 역공학 연구 동향 및 최적의 악성코드 통신 메시지 분석환경 구축 연구」, 정보보호학회논문지, 2021, pp.175-186.

요약: 이 논문은 악성코드가 주고받는 통신 프로토콜을 분석하는 역공학 연구를 정리한 논문이다. 네트워크 기반 방법과 실행 기반 방법을 나누어서 설명하고, 둘을 섞은 하이브리드 분석 환경을 제안하였다. 실제 분석 환경을 잘 만드는 것도 연구에서 중요하다는 점이 보였다.

느낀 점: 이 논문은 코드 자체보다는 통신 메시지 분석 쪽에 가까웠지만, 역공학이 단순히 파일만 보는 작업이 아니라 환경까지 같이 봐야 한다는 걸 보여줘서 의미가 있었다. 범위가 넓다는 점에서 오히려 전체 흐름을 이해하는 데 도움이 되었다.

인용 예시: 신강식 외(2021, pp.175-186)는 프로토콜 역공학의 성능 향상을 위해 네트워크 기반 분석과 실행 기반 분석을 혼합한 환경이 필요하다고 보았다.

출처: <https://www.kci.go.kr/kciportal/ci/sereArticleSearch/ciSereArtiView.kci?sereArticleSearchBean.artiId=ART002711181>

# 3. 간단한 정리

이번에 논문들을 찾아보면서 느낀 점은, 역공학이라는 말 안에 생각보다 여러 분야가 들어 있다는 것이었다. 어떤 논문은 ARM 바이너리 분석을 다루고, 어떤 논문은 메모리 오류 검출을 다루고, 또 어떤 논문은 자동 리프팅이나 역난독화를 다루고 있었다. 처음에는 전부 비슷해 보였는데, 하나씩 읽어 보니까 보는 관점이 조금씩 달랐다.

또 하나 느낀 점은, 디스어셈블이라는 게 단순 번역 작업은 아니라는 점이다. 값의 흐름, 제어 흐름, 데이터 구조, 난독화, 실행 환경 같은 것까지 같이 봐야 분석이 제대로 된다는 내용이 여러 논문에 공통적으로 보였다. 그래서 이 분야는 한 가지 기술만 알아서는 부족하고 여러 방법을 같이 써야 하는 분야라고 생각했다.
